# 题后总结

## 知识点
滑动窗口 + 哈希表（字符频次统计）

## 原始思路（定长滑动窗口）
1. 统计字符串 p 中每个字符的频次
2. 使用固定大小为 p.length 的滑动窗口在 s 上移动
3. 维护窗口内字符的频次，与 p 的频次对比
4. 当窗口内字符频次与 p 完全相同时，记录起始位置

## 优化思路（不定长滑动窗口）- 参考灵茶山艾府
1. **单个数组统计**：只用一个 `cnt` 数组记录 p 中字符的需求量（正数表示需要，负数表示多了）
2. **动态调整窗口**：右边界不断扩展，当某个字符数量为负时，左边界收缩直到恢复平衡
3. **巧妙判断匹配**：当窗口长度恰好等于 p.length 时，说明所有字符都匹配成功
4. **无需额外比较**：通过 `cnt[index] < 0` 自动触发窗口收缩，避免了显式的频次比较

## 优化对比
| 方面 | 原始方法 | 优化方法 |
|------|---------|---------|
| 数据结构 | 可能需要多个哈希表 | 单个数组 (26 个元素) |
| 窗口类型 | 定长窗口 | 不定长窗口 |
| 判断逻辑 | 需要比较频次表 | 通过长度判断 |
| 代码复杂度 | 较复杂 | 更简洁 |

## 关键点
1. `cnt` 数组初始化为 p 中各字符的需求量
2. 右边界进入时 `cnt[index]--`，左边界移除时 `cnt[index]++`
3. `while(cnt[index] < 0)` 循环确保窗口内不会有多余字符
4. 当 `right - left + 1 === p.length` 时，说明窗口内字符完全匹配

## 易错点
1. 理解 `cnt[index] < 0` 的含义：当前字符数量超过了需求
2. 窗口收缩时要正确恢复被移除字符的计数
3. 判断匹配成功的时机：窗口长度恰好等于 p.length

## 复杂度
- 时间复杂度：O(n)，n 为 s 的长度
- 空间复杂度：O(1)，固定 26 个字母的数组
